# PRD - تطبيق المتابعة الجماعية للأعمال الصالحة (عائلي)

PRD - تطبيق المتابعة الجماعية للأعمال الصالحة (عائلي)  
الإصدار: V1.2 (MVP + ملاحق تقنية)  
التاريخ: 2026-02-19  |  المنطقة الزمنية: Asia/Riyadh  
الملكية: فريق المنتج والتقنية (للاستخدام الداخلي)

## 1) ملخص المنتج
تطبيق للهاتف (iOS/Android) يتيح للعائلات متابعة الأعمال الصالحة بصورة جماعية مع إمكانية تخصيص الأهداف لكل فرد، وتحديث التقدم بسهولة، وعرض لوحة أداء جماعية (نِسب فقط افتراضيًا) مع رسائل تشجيعية دون ذكر أسماء.

## 2) مبادئ التصميم
- تشجيع بلا إحراج: لا أسماء في الرسائل التشجيعية ولا لوم ضمني.
- الخصوصية أولًا: النسبة فقط افتراضيًا، مع إخفاء التوقيت وعدم عرض تفاصيل إلا باختيار المستخدم/الإدارة.
- بساطة التسجيل: تحديث التقدم يجب أن يكون سريعًا جدًا (ثوانٍ معدودة) مع دعم “العدّاد” بالضغط.
- تصنيف واضح للعبادات: تصنيف الأعمال إلى (عبادة القلب / عبادة اللسان / عبادة البدن) لتسهيل الفهم والفرز.
- مرونة القياس: دعم أعمال ثنائية (تم/لم يتم) وعددية (عدد/صفحات/دقائق) ومزدوجة (صفحات + دقائق).
- قابلية التوسع: دعم مجموعات خاصة وعامة مع أدوات إدارة وحماية مناسبة.

## 3) المشكلة التي يحلها التطبيق
- انقطاع الالتزام بسبب غياب المتابعة والتحفيز المنتظم.
- المتابعة الحالية عبر محادثات أو جداول غير منظمة وغير قابلة للقياس.
- اختلاف طبيعة الأعمال الصالحة يتطلب نظام قياس وتكرار مرن.
- الحاجة لتخصيص الأهداف لكل فرد مع مشاركة تقدم مناسب للعائلة.

## 4) الأهداف ومؤشرات النجاح
أهداف المنتج:
- تمكين إنشاء/الانضمام لمجموعات عائلية بسهولة وإضافة الأعضاء.
- تمكين كل مستخدم من اختيار أعماله وأهدافه وتكراراته وإشعاراته.
- رفع معدل التسجيل اليومي للتقدم وتحسين الاستمرارية (Streak).
- تقديم لوحة جماعية مشجعة تعتمد على النِسب فقط افتراضيًا.

مؤشرات نجاح (KPIs):
- D1 و D7 Retention.
- Daily Active Log Rate (معدل تسجيل التقدم يوميًا).
- Goal Completion Rate (نسبة إكمال الأهداف).
- Group Activation Rate (تفاعل 60%+ من أعضاء المجموعة أسبوعيًا).
- Notification Effectiveness (الاستجابة بعد الإشعار خلال ساعتين).

## 5) نطاق الإصدار
### داخل نطاق V1 (MVP)
- تسجيل دخول: جوال أو بريد + OTP.
- مجموعات: خاصة/عامة + إدارة طلبات الانضمام (Public-Approval مستحسن افتراضيًا عند اختيار “عام”).
- قوالب أعمال + أعمال مخصصة (مع تصنيف نوع العبادة: قلب/لسان/بدن).
- تكرارات متنوعة + أهداف أسبوعية + نطاق زمني اختياري للعمل (تاريخ بداية/نهاية).
- الصلاة (الفرائض) + السنن كاختيارات.
- القرآن: صفحات/دقائق/كلاهما.
- تسجيل تقدم سريع + سجل نشاط + Offline Sync + Undo.
- وضع العدّاد التنازلي (Countdown) للأعمال العددية بالضغط (Tap Counter).
- لوحة شخصية + لوحة جماعية (نِسب فقط افتراضيًا) + إخفاء التوقيت.
- Grace Period لمدة 6 ساعات بإدارة Admin + رسائل تحفيزية كل ساعة أثناء Grace.
- رسائل تشجيعية دورية دون أسماء (Counts-only).
- تقارير بسيطة: أسبوعي + Streak.

### خارج نطاق V1 (لاحقًا)
- مصحف كامل/تلاوات داخل التطبيق (يمكن ربط لاحقًا).
- تكامل تلقائي لأوقات الصلاة حسب الموقع.
- نظام نقاط/مكافآت متقدم.
- لوحة ويب متقدمة لإدارة مؤسسية.

## 6) المستخدمون والأدوار
- Admin (مدير المجموعة): ينشئ المجموعة، يحدد وقت الإقفال Cutoff، ويدير فترة السماح Grace Period، ويتحكم بخيارات الرسائل التشجيعية.
- Member (عضو): يخصص أعماله وأهدافه، ويسجل تقدمه، ويتحكم بخصوصيته وإشعاراته.

## 7) المتطلبات الوظيفية
### 7.1 التسجيل والهوية (Authentication)
- تسجيل دخول برقم الجوال أو البريد الإلكتروني.
- إرسال OTP حسب قناة التسجيل (SMS للجوال / Email للبريد).
- ملف المستخدم: الاسم، صورة اختيارية، المنطقة الزمنية، إعدادات الإشعارات والخصوصية العامة.

معايير قبول:
- القدرة على إكمال التسجيل خلال أقل من دقيقة.
- نجاح OTP وإعادة الإرسال ضمن سياسات أمان (Rate limiting).

### 7.2 المجموعات (Groups) والظهور العام
- إنشاء مجموعة: اسم، وصف، صورة، نوع (عائلة افتراضيًا).
- أنماط الانضمام: خاص (دعوة/رمز)، عام مع موافقة (مستحسن افتراضيًا عند اختيار عام)، عام مفتوح.
- طلبات الانضمام للمجموعات العامة (Approval Queue) مع قبول/رفض من Admin.
- إدارة الأعضاء: إضافة/إزالة، تغيير دور، حظر، إبلاغ.

معايير قبول:
- Admin يستطيع تغيير نمط الظهور (خاص/عام) مع تحذير واضح.
- في نمط (عام مع موافقة) لا يصبح المستخدم عضوًا إلا بعد قبول Admin.

### 7.3 إعدادات Admin الحساسة: Cutoff + Grace + رسائل الساعة
- تحديد وقت إقفال اليوم (Cutoff Time) للمجموعة وفق منطقة المجموعة الزمنية.
- تفعيل Grace Period ثابتة لمدة 6 ساعات بعد وقت الإقفال.
- Admin يحدد “بداية فترة السماح” (وقت الإقفال) ويطبق أي تغيير من اليوم التالي لتجنب تضارب اليوم الحالي.
- تفعيل/تعطيل بث رسائل تشجيعية جماعية خلال Grace (اختياري).
- تفعيل رسائل تحفيزية شخصية كل ساعة خلال Grace لمن لم يغلق يومه السابق (افتراضيًا ON).

### 7.4 الأعمال الصالحة (Deeds) والقوالب + تصنيف العبادة
#### 7.4.1 تصنيف نوع العبادة (Worship Type)
يمكن للمستخدم عند إنشاء/تعديل عمل صالح اختيار نوع العبادة من:
- عبادة القلب
- عبادة اللسان
- عبادة البدن

ويستطيع:
- تسمية العبادة (Name)
- إضافة وصف مختصر (Description) لتحديد المقصود أو النية/الهدف
- اختيار طريقة القياس والتكرار كباقي الأعمال (تم/عدد/دقائق… إلخ)

#### 7.4.2 قوائم مقترحة (اختيارية) لكل نوع
عبادة القلب (اقتراحات):
- الإخلاص، المحبّة، الخوف، الرجاء، التوكّل، الإنابة، الصبر، الشكر، الخشوع

عبادة اللسان (اقتراحات):
- الذكر، تلاوة القرآن، الدعاء، الأمر بالمعروف والنهي عن المنكر، النصيحة الحسنة

عبادة البدن (اقتراحات):
- الصلاة، الصيام، الطواف

> ملاحظة تنفيذية: هذه القوائم تظهر كاقتراحات سريعة (Quick Add) ويمكن للمستخدم تعديل الاسم/الوصف بحرية.

#### 7.4.3 القوالب الافتراضية الأساسية (كما في V1)
- الصلوات الخمس + سنن/نوافل (اختيارات): الوتر، الضحى، قيام الليل، وغيرها.
- الأذكار: الصباح، المساء، بعد الصلاة، النوم، استغفار، صلاة على النبي.
- القرآن: قراءة (صفحات/دقائق/كلاهما).
- أعمال أخرى: صيام، صدقة، بر، صلة رحم.

خصائص كل عمل:
- نوع العبادة (قلب/لسان/بدن)
- الاسم، الوصف
- نوع القياس (Metric Type)
- الهدف (Target)
- التكرار (Recurrence)
- نافذة التحقق (للأذكار)
- نطاق زمني اختياري (Active From/To)
- الخصوصية، الإشعارات
- وضع العدّاد (إن كان العمل عدديًا)

### 7.5 القياس والتكرار (Metrics & Recurrence)
- أنواع القياس: تم/لم يتم، عدد، صفحات، دقائق، مزدوج (صفحات + دقائق).
- التكرار: يومي، أيام محددة، مرات في الأسبوع، هدف أسبوعي.
- نطاق زمني اختياري للعمل: تاريخ بداية/نهاية؛ خارج النطاق لا يظهر العمل في شاشة “اليوم” (مع إمكانية أرشفته).
- القرآن: يختار المستخدم (صفحات/دقائق/كلاهما). عند اختيار (كلاهما) تُعرض نسبتان منفصلتان افتراضيًا.

معايير قبول:
- عرض المتبقي اليوم/هذا الأسبوع بدقة حسب نوع التكرار.
- منع القيم غير المنطقية (قيم سالبة) مع خيار ضبط سقف منطقي لكل عمل مستقبلًا.

### 7.6 تسجيل التقدم (Progress Logging) + وضع العدّاد التنازلي
#### 7.6.1 طرق التسجيل القياسية
- زر (تم) للأعمال الثنائية.
- عدّاد +/– للأعمال العددية.
- إدخال مباشر (اختياري).
- Undo خلال نافذة قصيرة (مثال 30 ثانية) لتلافي الأخطاء.
- Offline-first: حفظ محلي ثم مزامنة.
- سجل نشاط للمستخدم (Private).

#### 7.6.2 وضع العدّاد التنازلي (Countdown Tap Counter)
ميزة اختيارية للأعمال ذات القياس العددي (Count) مثل الأذكار:
- يمكن للمستخدم تفعيل “وضع العدّاد” بزر داخل بطاقة العمل.
- عند التفعيل:
  - يبدأ “المتبقي” من قيمة الهدف اليومية (Target) وينخفض مع كل ضغطة.
  - في بداية “اليوم المنطقي” (حسب Cutoff للمجموعة) يعاد ضبط المتبقي تلقائيًا إلى Target.
  - كل ضغطة تسجل تقدّمًا (+1) ويحسب المتبقي = Target - Progress.
- عند الوصول للصفر:
  - يظهر “مكتمل” ويمكن (اختياريًا) السماح بالزيادة مع عرض “زيادة +N” دون أن يصبح المتبقي سالبًا.
- يدعم هذا الوضع كذلك شاشة “إغلاق اليوم السابق” أثناء Grace (يظهر المتبقي لليوم السابق مع نفس منطق العدّاد).

معايير قبول:
- تفعيل/إيقاف وضع العدّاد لا يغيّر سجل التقدم التاريخي؛ هو “عرض/طريقة إدخال” فقط.
- عند بدء اليوم المنطقي: تعود قيمة المتبقي لليوم الجديد إلى Target دون التأثير على بيانات الأيام السابقة.
- كل ضغطة تُسجل كتحديث واحد مع إمكانية Undo، وتعمل Offline ثم تُزامن.
- لوحة المجموعة لا تعرض توقيت الضغط أو عدد الضغطات، وتعرض النسبة فقط.

### 7.7 لوحة الأداء (Dashboards)
لوحة المستخدم (My):
- قائمة أعمال اليوم + المتبقي.
- Streak والتقدم الأسبوعي.
- وضع “إغلاق اليوم السابق” أثناء Grace (إن وُجد نقص).
- إظهار المتبقي للعداد التنازلي داخل بطاقة العمل.

لوحة المجموعة (Group):
- نِسبة إنجاز لكل عضو (افتراضيًا).
- بدون عرض توقيت أو “آخر تحديث”.
- لا تفاصيل أعمال إلا إذا فعّل Admin ذلك، مع بقاء تحكم المستخدم بخصوصيته.

### 7.8 الخصوصية (Privacy)
- مستويات لكل عمل: مخفي، نسبة فقط، تفاصيل.
- إخفاء “آخر وقت تحديث” دائمًا عن المجموعة (شرط ثابت).
- حساب نسبة لوحة المجموعة يعتمد فقط على الأعمال التي يشاركها المستخدم (غير المخفية).
- خيار إيقاف الظهور مؤقتًا (Pause) دون مغادرة المجموعة.

### 7.9 الإشعارات ورسائل التشجيع (Notifications & Encouragement)
الإشعارات الشخصية:
- تذكيرات مخصصة لكل عمل.
- تذكير إذا لم يتم التسجيل حتى ساعة محددة.
- وضع عدم الإزعاج (DND).
- رسائل Grace: تحفيز كل ساعة أثناء فترة السماح لمن لم يغلق يومه السابق (Private افتراضيًا).

رسائل تشجيعية جماعية (Counts-only، بدون أسماء):
- السابقون: أكملوا المطلوب اليوم وزيادة.
- المُنجزون: أكملوا 100%.
- المُثابرون: بين 40% و99%.
- على الطريق: أقل من 40% مع بداية تسجيل.

## 8) قواعد إقفال اليوم وفترة السماح (Grace Period Spec)
- لا يمكن تسجيل تقدم “أمس” عبر اختيار تاريخ أو تقويم.
- يحدد Admin وقت الإقفال (Cutoff) للمجموعة، ويبدأ اليوم الجديد فور وصول وقت الإقفال وفق منطقة المجموعة.
- خلال 6 ساعات بعد Cutoff: يظهر للمستخدم قسم “إغلاق اليوم السابق” مع عداد تنازلي، ويمكنه إكمال المتبقي لليوم السابق من هذا القسم فقط.
- لا يظهر للمجموعة أي توقيت للتحديث ولا أي تلميح أن التحديث حدث أثناء Grace.
- بعد انتهاء 6 ساعات: يُقفل اليوم السابق نهائيًا وتختفي واجهة الإغلاق المتأخر.

### 8.1 رسائل التحفيز كل ساعة أثناء Grace
- تُرسل كل ساعة (بحد أقصى 6 رسائل) للمستخدمين الذين لديهم يوم سابق غير مُغلق.
- تتوقف تلقائيًا عند إغلاق اليوم السابق أو عند تفعيل DND/إيقاف إشعارات Grace.
- لا تحتوي على أسماء أو إشارات للمجموعة، وتصل للمستخدم (Private) افتراضيًا.
- بث جماعي أثناء Grace: خيار Admin، وبصيغة Counts-only فقط.

### 8.2 معايير قبول (QA) - Grace
- عند Cutoff: ينتقل التطبيق لليوم الجديد، مع إتاحة “إغلاق اليوم السابق” فقط لمن لديهم نقص.
- ضمن (Cutoff -> Cutoff+6h): يستطيع المستخدم إكمال المتبقي لليوم السابق دون اختيار تاريخ.
- بعد (Cutoff+6h): أي محاولة تحديث لليوم السابق تُرفض برسالة واضحة.
- لوحة المجموعة لا تعرض توقيت ولا إشعار بأن التحديث متأخر.
- إشعارات كل ساعة تُرسل فقط لمن ينطبق عليهم الشرط وتتوقف فور الإغلاق.

## 9) المتطلبات غير الوظيفية (Non-Functional)
- الأداء: فتح شاشة اليوم خلال 2 ثانية على شبكة 4G.
- التوفر: 99.5% شهريًا مبدئيًا.
- الأمان: تشفير TLS أثناء النقل، وتشفير بيانات حساسة على الخادم.
- الموثوقية: Offline sync مع معالجة تضارب (آخر تحديث يفوز + سجل تعديل داخلي).
- اللغات: العربية أولًا، مع قابلية إضافة الإنجليزية لاحقًا.
- إمكانية الوصول: خطوط واضحة، دعم تكبير الخط، تباين مناسب.
- مقاومة الأخطاء في العدّاد: حماية من النقرات السريعة غير المقصودة + Undo سهل.

## 10) نموذج البيانات (High-Level Data Model)
- User(id, name, contact, timezone, settings)
- Group(id, name, visibilityMode, timezone, cutoffTime, graceEnabled, graceHours=6, settings)
- GroupMember(groupId, userId, role, joinedAt, memberSettings)

- DeedTemplate(id, worshipType, category, name, metricTypeDefault)
- UserDeed(id, userId, groupId, worshipType, name, description, metricType, target, recurrenceRule, activeFrom?, activeTo?, privacyLevel, reminders, counterMode?, allowOverTarget?)
- ProgressLog(id, userDeedId, logicalDay, valueCount/pages/minutes, createdAt, sourceOffline, note)

## 11) التحليلات (Analytics)
- Daily Active Log Rate.
- Retention (D1/D7).
- Completion Rate per Worship Type (قلب/لسان/بدن) + per Category.
- Effect of Grace reminders (hourly) on closing previous day.
- Usage of Counter Mode: نسبة الأعمال المفعّل لها عدّاد + معدل الضغطات.
- Group activation and churn.
- Notification engagement: open rate, action rate.

## 12) إدارة المجموعات العامة (Safety & Moderation)
- وضع (عام مع موافقة) يُنصح به افتراضيًا.
- زر إبلاغ عن مستخدم/محتوى + حظر من Admin.
- سياسات اسم/وصف المجموعة لمنع الإساءة (فلترة بسيطة + إبلاغ).
- عدم مشاركة أي بيانات حساسة: لا أرقام هواتف في واجهة المجموعة، ولا أسماء إلزامية.

## 13) خطة التسليم (Epics & Sprints)
Epics:
- A: Authentication & Onboarding
- B: Groups & Admin (Private + Public Approval)
- C: Deeds & Scheduling (including Worship Type + Quran dual + Active date range)
- D: Progress Logging (Offline, Undo, Counter Mode)
- E: Dashboards (My + Group ratios)
- F: Notifications & Encouragement (hourly Grace)
- G: Privacy & Moderation
تقسيم سبرنتات مقترح:
1. Sprint 1: Auth + Groups (Private) + Onboarding
2. Sprint 2: Deeds + Scheduling + Counter Mode + Progress Logging
3. Sprint 3: Dashboards + Privacy + Grace UI
4. Sprint 4: Notifications (hourly Grace) + Public Approval + Moderation + Analytics

## 14) خارطة الطريق (بعد V1)
- V1.2: قوالب موسمية (رمضان/عشر ذي الحجة)، خيار قضاء مضبوط، دعم الإنجليزية.
