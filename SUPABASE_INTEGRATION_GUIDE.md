# دليل دمج Supabase (Supabase Integration Guide)

## نظرة عامة
تم نقل الخلفية (Backend) للتطبيق بالكامل من Firebase إلى **Supabase**. يوفر Supabase كلاً من المصادقة (Authentication)، وقاعدة البيانات (PostgreSQL)، والأمان (Row Level Security - RLS).

تم إنشاء الجداول التالية:
- `profiles` (بيانات المستخدمين)
- `groups` (المجموعات)
- `group_members` (أعضاء المجموعات)
- `deeds` (الأذكار والأعمال الصالحة)
- `progress_logs` (سجلات التقدم)
- `reports` (تقارير المجموعات)

## 1. إعداد المت المتغيرات البيئية (Environment Variables)

لكي يعمل التطبيق مع مشروعك على Supabase، يجب عليك إنشاء ملف `.env` في المسار الرئيسي للمشروع وإضافة القيم الخاصة بك كالتالي:

```env
VITE_SUPABASE_URL=رابط_مشروعك_هنا
VITE_SUPABASE_ANON_KEY=المفتاح_الخاص_بك_هنا
```
*ملاحظة: يمكنك إيجاد هذه القيم في لوحة تحكم Supabase > Project Settings > API.*

## 2. المصادقة باستخدام الجوال (Phone Authentication)

تم تنفيذ وظيفة تسجيل الدخول عن طريق رقم الجوال (OTP). لكي تعمل هذه الميزة بدلاً من Firebase:
1. اذهب إلى لوحة تحكم مشروع Supabase.
2. انتقل إلى **Authentication** ثم **Providers**.
3. قم بتفعيل مزود خدمة الهاتف (Phone).
4. عليك ربط حسابك بأحد مزودي خدمة رسائل SMS (مثل Twilio أو MessageBird) وإضافة بيانات الاعتماد (Credentials) المطلوبة.

في حال أردت استخدام البريد الإلكتروني فقط في الوقت الحالي، فهو جاهز ومفعل افتراضياً في Supabase.

## 3. الدخول بواسطة Google وربط الحسابات (Google OAuth & Account Linking)

تمت إضافة إمكانية تسجيل الدخول وإنشاء حساب باستخدام حسابات Google بنجاح.

### سياسة ربط ودمج الحسابات (Account Merging Policy)
يتعامل Supabase مع مسألة ربط الحسابات بأمان. إذا حاول المستخدم تسجيل الدخول باستخدام Google وكان هناك حساب مسبق يمتلك نفس البريد الإلكتروني (عن طريق كلمة المرور):
- **لجعل النظام يقوم بدمج الحسابين تلقائياً** (بدون رسائل خطأ أو إنشاء حسابات مكررة)، يجب عليك تفعيل خيار **"Link identity to user"** من إعدادات المصادقة الخاصة بمشروعك في Supabase (لوحة التحكم > Authentication > Providers).
- بعد الربط، يستطيع المستخدم استخدام كلاً من (البريد/كلمة المرور) أو (تسجيل الدخول باستخدام جوجل) للوصول إلى نفس الحساب والبيانات.
- أثناء عملية الدخول، يتم تحديث جدول `profiles` تلقائياً عبر دالة (Trigger) لجلب اسم المستخدم وصورته الشخصية (`avatar_url`) في حال كانت فارغة، وذلك لضمان عدم الكتابة فوق بيانات معدلة مسبقاً. كافة الجداول المرتبطة مثل (التقارير وسجلات التقدم) تبقى كما هي دون أي تأثير لأنها تعتمد على مُعرّف `User ID` فريد.

### إعداد التطبيق في Google Cloud و Supabase
لكي يعمل زر تسجيل الدخول باستخدام Google، يجب عليك تنفيذ الخطوات التالية:

1. **في منصة Google Cloud Console**:
   - قم بالدخول إلى [Google Cloud Console](https://console.cloud.google.com).
   - أنشئ مشروعاً جديداً أو اختر مشروعاً حالياً.
   - اذهب إلى **APIs & Services** > **OAuth consent screen**، وحدد نوع المستخدمين كـ "External"، ثم املأ البيانات الأساسية (اسم التطبيق، البريد، إلخ).
   - انتقل إلى **Credentials** وانقر على **Create Credentials** ثم اختر **OAuth client ID**.
   - اختر نوع التطبيق **Web application**.
   - في خانة **Authorized redirect URIs**، أدخل رابط إعادة التوجيه الخاص بـ Supabase (والذي يكون عادةً بالصيغة: `https://[PROJECT_ID].supabase.co/auth/v1/callback`). 
   *يمكنك إيجاد هذا الرابط من لوحة تحكم Supabase > Auth > Providers > Google.*

2. **في لوحة تحكم Supabase**:
   - اذهب إلى **Authentication** > **Providers**، واختر **Google**.
   - فعّل الخيار (Enable Google provider).
   - أدخل الـ **Client ID** و **Client Secret** الذي حصلت عليهما من Google Cloud بمجرد إنشائك لبيانات الاعتماد (Credentials).
   - تأكد من تفعيل خيار **Link database identity to user** لدمج الحسابات كما ذكرنا سابقاً.
   - لحفظ الجلسات وتوجيه المتصفح، تأكد من إضافة رابط موقعك (مثل `http://localhost:5173` محلياً أو رابط الموقع الفعلي بعد نشره) داخل قسم **URL Configuration** تحت تبويب **Authentication** في Supabase.

### لاختبار Google Login:
- محلياً: يجب تشغيل التطبيق على الرابط الحقيقي الذي يدعمه رد توجيه Google. (عادةً `http://127.0.0.1:5173` أو الذي حددته في Authorized redirect).
- إنتاجياً: تأكد من إضافة رابط موقعك الحقيقي (Production URL) إلى الـ URL Configuration داخل Supabase وإلى Authorized redirect URLs داخل Google Cloud.

## 4. الأمان وقواعد (Row Level Security - RLS)

تم تفعيل قواعد RLS، ما يعني أن البيانات محمية بشكل تام، ولا يمكن للمستخدمين الوصول إلا للبيانات الخاصة بهم أو المجموعات التي ينتمون إليها. لا حاجة لأي إعداد إضافي من قبلك؛ جميع القواعد تم برمجتها داخل قاعدة البيانات.

## 5. وضع التجربة المؤقت للمصادقة (Temporary Auth Testing Mode)

تم تنفيذ وضع تجريبي لتسهيل اختبار التطبيق من قبل المستخدمين التجريبيين دون الحاجة لتأكيد البريد الإلكتروني أو تلقي رموز OTP عبر الجوال.

### كيف يعمل الوضع التجريبي:
- **معرّف موحد:** يمكن للمستخدم إدخال رقم الجوال (مثل `05XXXXXXXX`) أو البريد الإلكتروني في حقل واحد، مع كلمة المرور.
- **تخطي OTP للتجربة:** يتم تسجيل الدخول مباشرة عبر كلمة المرور حتى لأرقام الجوال (Supabase يدعم `signInWithPassword` برقم الجوال).
- **سهولة إنشاء الحسابات:** صفحة التسجيل تتيح إنشاء الحسابات مباشرة للاختبار.

### كيفية التفعيل / الإلغاء (Rollback Plan):
لتفعيل هذا الوضع يجب تعيين المتغير التالي في ملف `.env` (أو محلياً / في Vercel):
```env
VITE_AUTH_TEST_MODE=true
```

**للعودة للوضع الرسمي (إطلاق الإنتاج):**
1. قم بتغيير قيمة المتغير السابق أو إزالته تماماً:
   ```env
   VITE_AUTH_TEST_MODE=false
   ```
2. في لوحة تحكم Supabase > Authentication > Providers > Email، تأكد من تفعيل خيار **"Confirm email"** أو **"Secure email change"** إذا رغبت بفرض تأكيد الرسائل قبل الدخول.
3. عندها ستعود واجهة تسجيل الدخول لعرض خيارات (البريد الإلكتروني / رقم الجوال عبر OTP) بشكلها الرسمي المعتاد.

⚠️ **تحذير:** لا يُنصح بترك `VITE_AUTH_TEST_MODE=true` في بيئة الإنتاج النهائية (Production) إذا كنت ترغب في التحقق الكامل من هوية المستخدمين عبر OTP أو تأكيد البريد.

## 6. نقل البيانات من Firebase (Migration)

إذا كانت هناك بيانات قديمة موجودة على مشاريع Firebase، فلن يتم نقلها تلقائياً. سيتطلب الأمر كتابة سكريبت لتحميل البيانات من Firestore وإعادة كتابتها في جداول PostgreSQL الخاصة بـ Supabase باستخدام الـ API، مع مراعاة نقل الـ Auth Users أولاً لتفادي مشاكل الـ Foreign Keys.

## 7. تشغيل التطبيق محلياً
تأكد من تثبيت الحزم الجديدة:
```bash
npm install
```
للتشغيل محلياً:
```bash
npm run dev
```
